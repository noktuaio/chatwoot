## Resumo executivo

PR14 bundles 8 CRM "Pro" fronts. It is far too large for one PR and contains uneven risk: two fronts (kanban card UI, timeline humanize) are low-risk, high-value polish; three (AI summary, filters, reports) are medium-risk feature work with payload/migration/cost implications; two (intelligent handoff, permissions) are high-risk and gate or block others. Reviews graded 5 fronts SOLID and 3 NEEDS_WORK (intelligent-handoff, stage-as-label, crm-reports-dashboard, crm-permissions — handoff and permissions in particular carry correctness blockers).

Recommended split: 9 sub-PRs across 3 waves. WAVE A (foundations) lands the permissions layer (PR14.1) and a small shared backend-payload-hardening PR (PR14.2) first, because permissions gate every UI and the payload builders are touched by 4 fronts — doing them once prevents merge churn and the recurring "realtime-parity / visibility-leak" class of bug. WAVE B ships the high-value, low-coupling UX wins (card UI 14.3, timeline 14.4, AI summary 14.5, stage badge 14.6, filters 14.7). WAVE C ships the heavier, decision-gated work (reports 14.8, handoff 14.9a/9b) once the PO has answered open questions and the foundation PRs are merged.

Three things must be resolved before ANY build starts: (1) the permissions backfill/cutover decision (existing custom-role agents silently lose board access on deploy); (2) the EE namespace/prepend_mod_with wiring for Front 10 (as specified the policies no-op); and (3) the two correctness blockers in handoff (bot-keeps-replying after handoff; loop guard) and reports (no won/lost timestamp — KPIs would be computed off updated_at). Total realistic effort is ~135–175 engineering hours (~4–5 dev-weeks for one engineer; ~3 calendar weeks with two), excluding specs unless requested.

Net recommendation: APPROVE the split and Wave A + Wave B for immediate build; hold Wave C pending PO answers and the must-fix design corrections called out per front.

## Riscos transversais

- Shared payload builders touched by 4 fronts (CardPayloadBuilder, PayloadBuilder/DetailPayloadBuilder, ActivityPayloadBuilder). Mitigation: PR14.2 owns all additive board-payload + visibility changes; later fronts consume, never re-edit the same fields. Each builder change has exactly one owning PR.
- Realtime parity (cardMatchesFilters) is a recurring correctness trap: any server filter not mirrored client-side lets live upserts bypass it (root cause of the existing Status bug). Mitigation: PR14.2 centralizes the default-filters object and drops status from cardMatchesFilters; PR14.7 documents a per-filter predicate-vs-refetch contract; responsible=bot/ai-pending are server-only+refetch (cannot be evaluated client-side).
- Visibility / message-content leakage: metadata['ai'] (summary) and activity labels can serialize conversation content to agents who cannot see the conversation. Mitigation: PR14.2 adds metadata['ai'] sanitization to sanitize_hidden_conversation! and gates ai_summary on primary_conversation_visible?; PR14.4 enriches labels only over the already-visible/sanitized activity set; new endpoints (14.6 stage badge, 14.5 summarize) reuse authorize_crm_conversation!/AccessAuthorizer.
- OpenAI cost: both AI summary (14.5) and handoff (14.9) call the model. Mitigation: summary short-circuits on cache+freshness+unchanged-last-message and is gated to editors (crm_manage_ai); handoff extends the SINGLE existing classification call (no second round-trip) plus an idempotency guard so re-evaluation on every message doesn't re-spend. Validate classification accuracy is unchanged after adding the handoff schema field.
- Permissions gate everything: if PR14.1's EE modules are mis-namespaced or the OSS policies lack prepend_mod_with, the whole layer silently no-ops; if the backfill is skipped at deploy, agents are locked out. Mitigation: ancestors-include smoke spec, mandatory hook on all 9 policies, runbook-sequenced backfill.
- Inline :style for dynamic stage hex color violates the stated 'no inline styles' rule but matches established in-repo precedent (CrmKanbanPage:982, Avatar.vue). Mitigation: follow precedent (inline backgroundColor dot/rail with slate fallback), document the pre-existing exception in PR descriptions, and add a min-contrast ring for pale colors on dark surfaces. Applies to 14.3 and 14.6.
- pt-BR-first deployment vs en-only i18n rule: new English keys won't appear translated in production until the locale file is updated out-of-band. Mitigation: PO decision per front on whether new strings must also land in pt_BR locale; note the existing timeline strings are actually English (Front 6 design misread them as pt-BR).
- Handoff bot-stop correctness: assigning a human does NOT stop the active inbox AgentBot from replying — without conversation.bot_handoff! the customer gets human+bot double answers. This is a Wave-C blocker isolated to 14.9a.

### FRONT 3 — Kanban CARD UI redesign (CrmKanbanCard.vue)
**Review:** `SOLID` · esforço design `M` / revisado `M` · sub-PR: Yes — its own focused sub-PR. It is self-contained (one component rebuild + one small pill component + a prop wire-up in the page + minor i18n + a ~5-line additive backend payload tweak), has no behavioural coupling to the drawer/AI/automation fronts, and is easy to review/revert visually. Title e.g. "feat(crm): redesign kanban card UI". Should land after (or alongside) the tiny backend payload additions but those can ride in the same PR since they're additive.

**Estado atual:** The card is a single `<button>` in `app/javascript/dashboard/routes/dashboard/crm/components/CrmKanbanCard.vue` (138 lines). Markup today: a `bg-n-surface-1` rounded box with (1) title + contact label row with a priority chip pinned right; (2) optional 2-line description; (3) optional AI-suggestion pill ("AI: {stage}"); (4) a meta row with responsible (text only, bot gets an `i-lucide-bot`) and value; (5) a footer row with inbox name + entered_stage date (pt-BR formatted inline via Intl). 

Problems found by reading the code:
- No avatar anywhere — responsible is plain text, so cards read as a wall of grey text. Visual hierarchy is flat (title/contact/responsible all `text-n-slate-11/12`, only weight differs).
- Priority chip is a filled coloured block that competes with everything for attention and is shown for EVERY card (default medium), creating noise rather than signal.
- Stage colour (`stage.color`) is only a 2.5px dot in the COLUMN header (CrmKanbanPage.vue line 980-983); the card itself carries zero stage identity, so a detached/dragged card has no colour cue.
- AI suggestion, when present, is a full-width blue pill that pushes layout down and looks like a second priority chip.
- Channel/inbox is shown as raw text name only — no channel icon, even though `inbox.channel_type` is already in the payload and a `ChannelIcon` component exists.
- `last_message_at` and `next_follow_up_at` are in the payload but NOT rendered at all — the two most actionable CRM signals (recency, follow-up due) are invisible.
- `score` (0-100, AI lead score) is in the payload, unused on the card.
- Dates are hand-formatted with `Intl.DateTimeFormat('pt-BR', …)` inline (hardcoded locale, absolute timestamp, not relative "2d ago").
- Standalone variant just falls back to a text label ("Standalone card"); no distinct visual treatment.
- No hover/drag affordance beyond `hover:bg-n-alpha-2`; the column sets `ghost-class="opacity-40"` and `drag-class="cursor-grabbing"` but the card has no grip handle, no elevation-on-hover, no quick-open affordance.

Data available in board payload (confirmed in `app/services/crm/kanban/card_payload_builder.rb` + board_payload_builder.rb): id, title, description (note: description is NOT in CardPayloadBuilder identity/status/link payloads — it renders on the card today but is absent from the board builder; see Risks), value_cents, currency, priority, score, status, is_standalone, next_follow_up_at (iso8601), last_message_at (raw ts), entered_stage_at (raw ts), ai_suggestion{id,to_stage_id,to_stage_name,confidence}, contact{id,name,phone_number}, owner, responsible{type:'agent'|'bot',id,name} (derived realtime, NO avatar url), inbox{id,name,channel_type}, conversation{id,display_id,status}. Stage color/name live in the parent stage payload, NOT per-card.

Reusable in-house building blocks confirmed: `components-next/avatar/Avatar.vue` (props name/src/size/iconName/inbox, renders initials + colour-by-name + optional channel badge), `components-next/icon/ChannelIcon.vue` (takes `inbox`), `components-next/Conversation/ConversationCard/ConversationCard.vue` already does avatar + channel badge + time-ago meta row and is the closest existing precedent to mirror.

**Proposta:** Redesign CrmKanbanCard as a denser, scannable card with a left stage-colour accent rail and a clear 3-zone hierarchy, mirroring Pipedrive/Attio deal cards while staying in Chatwoot tokens. Keep it a single `<button>` (preserves current open-on-click + draggable wrapper).

ZONES (top→bottom):
1. Left accent rail: a 3px full-height bar in the stage colour (`stage.color`) so each card carries its stage identity even mid-drag. Pass `stageColor` as a new prop from CrmKanbanPage (`v-for stage` already in scope: `<CrmKanbanCard :card="element" :stage-color="stage.color" … />`). Render as an absolutely-positioned `before`-style div or a flex child; use inline arbitrary value via Tailwind `bg-[…]` is not possible with dynamic value, so use a thin `<span>` with `:style` ONLY for the dynamic colour — note this technically conflicts with the "no inline styles" rule, BUT the existing column header already does exactly this (`:style="{ backgroundColor: stage.color }"`), so it is the established project pattern for stage colour. Flag for owner (Open Question 1).
2. Header row: contact/responsible Avatar (size 24, `roundedFull` for people, square for bot via `iconName="i-lucide-bot"`) + title (truncate, `text-sm font-medium text-n-slate-12`) on line 1, contact name/phone secondary line (`text-xs text-n-slate-11`). Priority is demoted from a filled chip to a small coloured dot before the title (only rendered for high/urgent; low/medium = no dot) so priority becomes signal not noise.
3. Signal row (the new value): horizontal wrap of compact pills/icons, each shown only when data present — channel icon (ChannelIcon from inbox), value (currency, `font-medium text-n-slate-12`), follow-up due (`i-lucide-clock` + relative time, colour-shifts to `n-ruby` when overdue using next_follow_up_at < now, `n-amber` when due ≤24h), AI suggestion badge (compact `i-lucide-sparkles` + stage name, `bg-n-blue-3 text-n-blue-11`, with confidence-driven tone: ≥0.75 solid, 0.55-0.75 subtle), score (only if >0, small `n-teal` chip). 
4. Footer row: responsible name (`text-[11px] text-n-slate-10`) on the left, last-message recency on the right (relative time from last_message_at; if stale >Xd, dim). entered_stage_at moves to a title/tooltip rather than always-on text to cut clutter.

Density: target ~88-104px tall for a typical linked card, collapsing rows that have no data (standalone card with only a title should be ~56px). Use `gap-2`/`gap-1.5`, `p-3` padding kept.

PERMISSION KEYS (7, namespaced `crm_*`):
- `crm_view` — see the CRM module: sidebar entry, route access, board read, card drawer read (read-only). Base capability; all other crm_* imply view in the UI.
- `crm_manage_cards` — create/edit/delete cards, link/unlink conversations & contacts, manage follow-ups.
- `crm_move_cards` — drag a card between stages (the high-frequency, lower-trust action; separated so you can grant "look + drag" without full card edit). move? requires crm_move_cards OR crm_manage_cards.
- `crm_manage_pipelines` — create/edit/delete pipelines, stages (incl. reorder), stage automations & steps, pipeline↔inbox mapping, inbox settings. (The "admin config" of the funnel.)
- `crm_manage_ai` — edit AI settings panel (models/thresholds/auto-move) and trigger per-card AI evaluation/accept/dismiss suggestions.
- `crm_view_reports` — (forward-looking) view CRM reports/analytics once that front ships. Ship the key now so roles don't need re-editing later; it gates nothing until reports exist.
- `crm_admin` — umbrella convenience that implies all of the above (UI auto-selects children when checked, mirroring the existing `conversation_manage` → unassigned+participating cascade in CustomRoleModal.vue). Backend treats crm_admin as "includes every crm_* key".
